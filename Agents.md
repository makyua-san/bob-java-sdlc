# Agents.md — AI エージェント設定ファイル

> このファイルは、生成AIがプロジェクトのコンテキストを正しく理解し、
> 各工程で適切なドラフトを生成するための前提条件を定義します。
> **すべてのAIエージェントはこのファイルを最初に参照してください。**

---

## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| プロジェクト名 | 備品購入申請アプリ |
| 目的 | 社内の備品購入申請・承認フローをシステム化する |
| 技術スタック | Java 21 / Spring Boot / Maven |
| アーキテクチャ | RESTful API + サーバーサイドレンダリング |
| インフラ | IaC（Terraform）によるクラウド環境構成 |
| 言語 | **コード中のコメント、コミットメッセージ、ドキュメント、テストメソッド名はすべて日本語で記述すること** |

### MVP機能
- 申請者が備品購入申請を登録できる
- 承認者が承認 / 却下できる
- 申請一覧を確認できる
- 却下時は理由を必須にする
- 高額申請（5万円超）には注意表示を出す

### 主要業務ルール
- 高額申請の閾値は **設定値として外出し** する（ハードコーディング禁止）
- 却下時の理由は必須（空文字・null不可）
- 承認 / 却下は承認権限を持つユーザーのみ実行可能
- 申請のステータス遷移: `下書き → 申請済み → 承認済み / 却下`（逆方向遷移は不可）

## 2. 開発プロセスの制約

### 仕様駆動の原則

AIは以下の順序を厳守してドラフトを生成すること。**順序を飛ばした依頼があった場合は、前工程の成果物の存在を確認するよう促すこと。**

```
仕様（requirements / acceptance criteria）
  → 設計（アプリ設計 / 配備設計 / 観測設計）
    → テスト設計（テストシナリオ / テストケース）
      → 実装（コード / 単体テスト）
        → レビュー（多層レビュー）
```

### 改修時の原則

既存機能に対する改修依頼を受けた場合、AIは以下を遵守すること:

1. **いきなりコードを修正しない。** まず仕様差分を整理する
2. 仕様差分 → 設計差分 → テスト差分 → 実装差分 の順で進める
3. 差分を作成する際は、必ず既存の成果物（requirements.md, design-app.md 等）を参照し、既存内容との整合性を確認する
4. 変更による **既存機能への影響範囲** を明示する

### 成果物体系

| 工程 | 成果物 | 改修時の差分版 |
|---|---|---|
| 仕様 | `requirements.md`, `acceptance-criteria.md` | `requirements-v{N}.md`, `acceptance-criteria-v{N}.md` |
| 設計（アプリ） | `design-app.md` | `design-app-v{N}.md` |
| 設計（配備） | `design-deployment.md` | `design-deployment-v{N}.md` |
| 設計（観測） | `design-observability.md` | `design-observability-v{N}.md` |
| テスト | `test-scenarios.md` | `test-scenarios-v{N}.md` |
| トリアージ | `change-triage.md` | — |
| インフラ | `infra/` | — |

## 3. 役割と期待するAIの振る舞い

> 生成AIは独立した役割ではなく、以下の各ペルソナが自分の業務でAIを活用するツールである。
> AIは **依頼者の役割に応じた視点** でドラフトを生成し、他の役割のレビュー観点も補足として提示すること。

### 業務担当者として依頼された場合
- 業務要望を構造化し、ユーザーストーリーとして整理する
- 業務用語を正確に使い、技術用語への安易な置き換えを避ける
- 曖昧な表現があれば、確認すべき点を明示する

### PO / 業務責任者として依頼された場合
- スコープの妥当性、優先順位、受け入れ条件の観点でレビューする
- 改修トリアージでは、業務価値・実装コスト・テスト影響・運用影響の比較表を作成する
- 仕様の過不足を指摘し、MVP範囲を明確にする

### 開発者として依頼された場合
- 仕様と設計に基づいたコードを生成する（後述のJavaコーディング規約に準拠）
- 設計ドラフトでは、アプリ設計・配備設計・観測設計の3観点を必ず含める
- 実装可能な粒度か、曖昧さがないかを確認し、不明点は質問として明示する
- ログ出力は後付けではなく、最初から実装に含める

### QA担当として依頼された場合
- テストケースは正常系・異常系・境界値・権限制御・回帰テストを網羅する
- 各テストケースに「このテストで何を確認するか」の目的を明記する
- 仕様の曖昧さや穴を発見した場合は、テストケースとともに指摘事項として明示する
- 境界値は具体的な数値で示す（例: 「5万円超」→ 50,000円は含まない、50,001円から該当）

### 運用担当として依頼された場合
- IaC雛形、デプロイ手順、チェックリストを生成する
- 観測設計では、構造化ログ項目・メトリクス・アラート候補・トラブル切り分け観点を含める
- 運用影響（ログ量増加、メトリクス追加、アラート閾値変更）を明示する

## 4. 多層レビュー時のAIの振る舞い

このプロジェクトでは、4層のレビュープロセスを採用する。AIが関与する層では以下の観点でレビューすること。

### 第1層: 開発者ローカル AI レビュー

開発者から直接レビューを依頼された場合、以下を確認する:
- 仕様（requirements.md / acceptance-criteria.md）との整合性
- 設計（design-app.md）との整合性
- 命名規約の遵守（本ファイルのコーディング規約を参照）
- 責務分離（Controller / Service / Repository の責務が混在していないか）
- 例外処理の妥当性（カスタム例外の使用、エラーレスポンスの統一）
- ログ出力の漏れ（design-observability.md のログ項目と照合）
- テスタビリティ（DIが適切か、モック可能な構造か）
- 高額判定の閾値など、設定値がハードコーディングされていないか

### 第3層: PR AI レビュー

PRの差分全体に対して、以下を確認する:
- 変更による既存機能への影響範囲
- 仕様差分・設計差分との整合（差分成果物が存在する場合）
- テストカバレッジの不足（acceptance-criteria に対応するテストがあるか）
- ロジックの重複（同じ判定が複数箇所に存在していないか）
- 回帰リスクの指摘

## 5. 設計ドラフトの生成ルール

AIが設計ドラフトを生成する際は、以下の3観点を **必ず** 含めること。

### アプリ設計
- ドメインモデル（エンティティ、値オブジェクト、関連）
- API設計（エンドポイント、メソッド、リクエスト/レスポンス）
- ステータス遷移（状態遷移図または遷移表）
- バリデーション設計（入力検証ルール一覧）

### 配備設計
- 実行環境（ランタイム、コンテナ構成）
- IaC対象（Terraform管理するリソース一覧）
- 設定値管理（環境変数、プロファイル分離）
- DB接続（接続プール設定、マイグレーション方針）

### 観測設計
- 構造化ログ項目（出力項目: requestId, userId, action, timestamp, 業務データ）
- メトリクス（レスポンスタイム、エラー率、業務KPI）
- アラート候補（閾値、通知先、エスカレーション）
- トラブル切り分け観点（ログの追跡手順、典型的な障害パターン）

## 6. Java 開発環境

### 必須環境

| 項目 | バージョン |
|---|---|
| Java | 21 |
| Maven | 3.9系 |
| Spring Boot | 3系 |

### プロジェクト構成

```
src/
├── main/
│   ├── java/
│   │   └── com/
│   │       └── example/
│   │           └── purchaserequest/
│   │               ├── controller/    # REST APIエンドポイント
│   │               ├── service/       # ビジネスロジック
│   │               ├── repository/    # データアクセス層
│   │               ├── model/         # エンティティ・DTO
│   │               │   ├── entity/    # DBエンティティ
│   │               │   └── dto/       # リクエスト・レスポンスDTO
│   │               ├── config/        # 設定クラス（閾値等の外部設定含む）
│   │               ├── exception/     # カスタム例外
│   │               └── util/          # ユーティリティクラス
│   └── resources/
│       ├── application.yml            # アプリケーション設定
│       ├── application-dev.yml        # 開発環境設定
│       ├── application-prod.yml       # 本番環境設定
│       └── logback-spring.xml         # ログ設定（構造化JSON出力）
└── test/
    ├── java/
    │   └── com/
    │       └── example/
    │           └── purchaserequest/
    │               ├── controller/    # コントローラーテスト
    │               ├── service/       # サービステスト
    │               └── repository/    # リポジトリテスト
    └── resources/
        └── application-test.yml       # テスト環境設定
```

### コーディング規約

**命名規則:**
- クラス名: PascalCase（例: `PurchaseRequestService`, `ApprovalController`）
- メソッド名: camelCase（例: `getRequestById`, `approveRequest`）
- 定数: UPPER_SNAKE_CASE（例: `HIGH_AMOUNT_THRESHOLD`, `DEFAULT_PAGE_SIZE`）
- パッケージ名: 小文字のみ（例: `com.example.purchaserequest.service`）

**コード品質:**
- 1メソッドは50行以内を目安に
- 1クラスは500行以内を目安に
- 循環的複雑度は10以下を維持

**コメント:**
- Javadocを使用してpublicメソッドとクラスを文書化（日本語）
- 複雑なロジックには日本語でインラインコメントを追加
- TODOコメントには担当者と期限を記載

### 依存性管理

**Maven設定:**
- `pom.xml`でバージョンをプロパティで管理
- 依存関係の衝突を避けるため`<dependencyManagement>`を活用
- 不要な依存関係は定期的に削除

**推奨ライブラリ:**
- Spring Boot: エンタープライズアプリケーション開発
- Lombok: ボイラープレートコード削減
- MapStruct: オブジェクトマッピング
- JUnit 5: ユニットテスト
- Mockito: モックテスト
- AssertJ: アサーション

### エラーハンドリング

**例外処理:**
- チェック例外は必要最小限に
- カスタム例外クラスを作成して意味のある例外を投げる
- `@ControllerAdvice`でグローバル例外ハンドラーを実装

**ログ出力:**
- SLF4J + Logbackを使用
- **構造化ログ（JSON形式）** を標準とする
- ログレベルを適切に設定（ERROR, WARN, INFO, DEBUG, TRACE）
- 本番環境では個人情報をログに出力しない
- 全リクエストに `requestId` を付与し、追跡可能にする

### セキュリティ

**認証・認可:**
- Spring Securityを使用
- MVPでは簡易認証を使用し、将来的に外部IdP連携を想定する
- 承認操作は承認権限を持つユーザーのみ実行可能にする

**入力検証:**
- Bean Validationを使用（`@Valid`, `@NotNull`, `@Size`等）
- SQLインジェクション対策としてPreparedStatementを使用
- XSS対策として入力値をサニタイズ

### パフォーマンス

**データベース:**
- N+1問題を避けるため適切にJOINまたはFETCH JOINを使用
- インデックスを適切に設定
- コネクションプールを最適化（HikariCP推奨）

**キャッシング:**
- Spring Cacheを活用（`@Cacheable`, `@CacheEvict`）

### テスト

**テスト戦略:**
- ユニットテスト: カバレッジ80%以上を目標
- 統合テスト: 主要なビジネスフローをカバー
- E2Eテスト: クリティカルパスを検証

**テストコード品質:**
- Given-When-Thenパターンを使用
- テストメソッド名は **日本語で記述** する（例: `高額申請の閾値を超える場合に注意表示が出ること`）
- テストデータはファクトリーパターンで管理

### 静的解析（SAST）

以下のツールをローカルおよびCI上で実行する:
- **SpotBugs**: バグ検出
- **Checkstyle**: コーディング規約チェック
- **OWASP Dependency-Check**: 依存ライブラリの既知脆弱性チェック

### CI/CD

**ビルド:**
- `mvn clean install`でビルドとテストを実行
- Mavenラッパー（`mvnw`）を使用してバージョン統一

### バージョン管理

**Git運用:**
- ブランチ戦略: Git Flow
- コミットメッセージ: 日本語で簡潔に記述
- プルリクエスト: 多層レビュー（AI + 人間）必須

### API仕様

- OpenAPI (Swagger)でREST API仕様を自動生成
- エンドポイントには日本語で説明を記載
