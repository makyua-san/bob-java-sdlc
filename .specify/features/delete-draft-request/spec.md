# 機能仕様: 下書き削除API

## 1. 概要

### 1.1 機能名
下書き申請削除機能

### 1.2 目的（WHY）
申請者が誤って作成した下書きや、不要になった下書きを削除できるようにすることで、申請一覧の整理と管理性を向上させる。下書き状態の申請は正式に提出されていないため、削除しても業務フローに影響を与えない。申請者が自分で不要な下書きを削除できることで、システムの使い勝手が向上し、データの整合性も保たれる。

### 1.3 対象ユーザー
- **申請者**: 自分が作成した下書き状態の申請を削除したいユーザー

### 1.4 関連仕様
- [`spec.md`](../../../spec.md): 備品購入申請システム（MVP）の基本仕様
- [`design-app.md`](../../../design-app.md): アプリケーション設計書

## 2. 機能要件（WHAT）

### 2.1 下書き削除API

#### 2.1.1 削除対象
- **削除可能な申請**: 下書き（DRAFT）状態の申請のみ
- **削除方式**: 論理削除（削除フラグによる管理）
- **削除後の扱い**: 削除された申請は一覧に表示されないが、データベースには保持される

#### 2.1.2 削除条件
以下の条件を**すべて**満たす場合のみ削除可能：

1. 申請のステータスが `DRAFT`（下書き）である
2. 削除を実行するユーザーが申請者本人である
3. 申請が既に削除されていない（論理削除済みでない）

#### 2.1.3 削除操作
- **HTTPメソッド**: `DELETE`
- **エンドポイント**: `/api/v1/requests/{id}`
- **認証**: 必須（ログインユーザーのみ）
- **権限**: 申請者本人のみ（承認者でも他人の申請は削除不可）

#### 2.1.4 削除時の処理
1. 申請IDで申請を取得
2. 申請が存在しない場合はエラー（404 Not Found）
3. 申請者本人でない場合はエラー（403 Forbidden）
4. ステータスがDRAFT以外の場合はエラー（400 Bad Request）
5. 既に削除済みの場合はエラー（400 Bad Request）
6. 削除フラグを立てる（`deleted = true`, `deletedAt = 現在時刻`, `deletedBy = ユーザー名`）
7. 成功レスポンスを返す

### 2.2 論理削除の実装

#### 2.2.1 削除フラグ
`PurchaseRequest` エンティティに以下のフィールドを追加：

- `deleted`: Boolean型、デフォルト `false`
- `deletedAt`: LocalDateTime型、削除日時（nullable）
- `deletedBy`: String型、削除者のユーザー名（nullable）

#### 2.2.2 削除済み申請の扱い
- 申請一覧取得時: `deleted = false` の申請のみ取得
- 申請詳細取得時: 削除済み申請にアクセスした場合はエラー（404 Not Found）
- 削除済み申請の復元: MVP範囲外（将来的な拡張として検討）

### 2.3 エラーハンドリング

#### 2.3.1 エラーケース

| エラーケース | HTTPステータス | エラーコード | エラーメッセージ |
|------------|--------------|------------|----------------|
| 申請が存在しない | 404 | REQUEST_NOT_FOUND | "申請が見つかりません" |
| 申請者本人でない | 403 | FORBIDDEN | "自分の申請のみ削除できます" |
| ステータスがDRAFT以外 | 400 | INVALID_STATUS | "下書き状態の申請のみ削除できます" |
| 既に削除済み | 400 | ALREADY_DELETED | "この申請は既に削除されています" |

## 3. 非機能要件

### 3.1 パフォーマンス
- 削除操作: 1秒以内
- 削除後の一覧取得: 既存の一覧取得と同等（1秒以内）

### 3.2 セキュリティ
- 認証: ログインユーザーのみ削除可能
- 認可: 申請者本人のみ削除可能（他人の申請は削除不可）
- 監査ログ: 削除操作を構造化ログに記録（requestId, userId, requestId, action: "delete_draft"）

### 3.3 データ整合性
- 論理削除により、削除履歴を保持
- 削除日時と削除者を記録し、監査可能にする
- 物理削除は行わず、データの復元可能性を確保

### 3.4 保守性
- 削除フラグによる管理で、既存のクエリへの影響を最小化
- 削除済み申請のフィルタリングはリポジトリ層で一元管理

## 4. 制約事項

## 7. データモデル変更

### 7.1 PurchaseRequestエンティティの変更

**追加フィールド**:
```java
@Column(nullable = false)
private Boolean deleted = false; // 削除フラグ

@Column
private LocalDateTime deletedAt; // 削除日時

@Column(length = 50)
private String deletedBy; // 削除者
```

**削除メソッド**:
```java
/**
 * 申請を論理削除
 * @param username 削除者のユーザー名
 */
public void delete(String username) {
    if (status != RequestStatus.DRAFT) {
        throw new IllegalStateException("下書き状態の申請のみ削除できます");
    }
    if (deleted) {
        throw new IllegalStateException("この申請は既に削除されています");
    }
    this.deleted = true;
    this.deletedAt = LocalDateTime.now();
    this.deletedBy = username;
    this.updatedAt = LocalDateTime.now();
}
```

### 7.2 データベースマイグレーション

## 10. 用語定義

| 用語 | 定義 |
|------|------|
| 論理削除 | データベースから物理的に削除せず、削除フラグを立てることでデータを非表示にする方式 |
| 削除フラグ | 削除されたかどうかを示すBoolean型のフィールド（deleted） |
| 削除日時 | 削除操作が実行された日時（deletedAt） |
| 削除者 | 削除操作を実行したユーザーのユーザー名（deletedBy） |
